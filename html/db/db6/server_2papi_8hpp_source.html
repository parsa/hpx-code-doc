27  // Remarks:
28  // As of papi-4.2.0 using bundled pfmlib many PAPI functions are
29  // subtly broken when accessing performance counter interface in Linux
30  // kernel.
31  // To maintain the HPX notion of single monitored quantity per counter
32  // instance, the events need to be added (and removed if counter "stop"
33  // function is to be reliably implemented) individually.
34 
35  // What has been tried so far:
36  // 1) Multiple PAPI event sets with one event per set are not properly
37  // coordinated by the library. Probably the only way this can work is when
38  // at most one event set per thread is active at a time, but that doesn't
39  // provide the necessary functionality.
40  // 2) Using a shared event set with PAPI_add_event works, but the removal
41  // via PAPI_remove_event causes the remaining counter(s) to stop counting,
42  // even though the event set status is indicated as PAPI_RUNNING.
43  // 3) Shared event set that is initialized from scratch by explicitly
44  // adding all remaining events when one of the existing events needs to be
45  // removed seems to work. However, the collected counts will be lost
46  // during event set cleanup. Maintaning the correct total counts requires
47  // careful tracking of values at both PAPI library and HPX levels.
48  // 4) To simplify (3), in theory one could use PAPI_write to reinstate
49  // the counter values before resuming counting. PAPI_write doesn't seem
50  // to be working on all platforms and in addition some kernels may
51  // explicitly disable it for accesses from user space.
52  // 5) The alternative to using PAPI_read is to invoke PAPI_accum to
53  // accumulate the momentary counts in user space counter array, with the
54  // side effect of resetting the values of PAPI counters. This simplifies
55  // the handling of counter values when adjustments to event set need to be
56  // performed. Unfortunately, just like for (2) the simple removal of events
57  // does not seem to work correctly, so sticking with (3) for event set
58  // shrinking is the preferred approach.
